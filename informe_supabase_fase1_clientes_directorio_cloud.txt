INFORME SUPABASE - FASE 1 CLIENTES / DIRECTORIO CLOUD
Proyecto: odlrhijtfyavryeqivaa
Fecha: 2026-06-23

OBJETIVO
Se implemento la FASE 1 para sincronizar clientes/directorio en licencias PRO con cloud_pos_sync activo. FREE queda local. Supabase queda como fuente oficial de verdad para clientes PRO. Dexie queda como cache local/offline. No se implementaron productos, caja, ventas, reportes ni abonos cloud.

TABLA CREADA
public.pos_customers
RLS habilitado. Sin politicas directas de lectura/escritura. Sin grants directos para anon/authenticated. Acceso por RPCs security definer.

COLUMNAS
id text primary key
license_id uuid not null references public.licenses(id) on delete cascade
name text not null
phone text null
phone_key text null
address text null
debt numeric not null default 0
debt_cents integer not null default 0
credit_limit numeric not null default 0
created_at timestamptz not null default now()
updated_at timestamptz not null default now()
deleted_at timestamptz null
server_version integer not null default 1
created_by_device_id uuid null references public.license_devices(id)
updated_by_device_id uuid null references public.license_devices(id)
created_by_staff_user_id uuid null references public.license_staff_users(id)
updated_by_staff_user_id uuid null references public.license_staff_users(id)
last_idempotency_key text null
metadata jsonb not null default '{}'

INDICES
idx_pos_customers_license_updated_at: (license_id, updated_at)
idx_pos_customers_license_server_version: (license_id, server_version)
idx_pos_customers_license_deleted_at: (license_id, deleted_at)
idx_pos_customers_license_name: (license_id, name)
idx_pos_customers_license_phone_key: (license_id, phone_key)
ux_pos_customers_license_phone_key_active: unique (license_id, phone_key) where phone_key is not null and deleted_at is null

CONSTRAINTS
Nombre no vacio.
Debt, debt_cents y credit_limit no negativos.
server_version mayor o igual a 1.
FKs hacia licenses, license_devices y license_staff_users.

FUNCIONES PRIVADAS
private.normalize_pos_customer_phone_key: normaliza telefono quitando caracteres no numericos.
private.pos_customer_to_jsonb: serializa cliente cloud a JSONB seguro.
private.assert_cloud_pos_sync_enabled: valida feature cloud_pos_sync.
private.insert_pos_idempotency_processing: registra llave idempotente en processing.
private.complete_pos_idempotency: guarda respuesta final y marca completed.

RPCs PUBLICAS
public.pos_upsert_customer:
- valida licencia, dispositivo, token del dispositivo, sesion staff si aplica y feature cloud_pos_sync.
- valida permiso customers para staff.
- crea o edita clientes.
- normaliza name, phone, phone_key, address y credit_limit.
- permite conservar deuda solo al crear/migrar; no muta deuda en updates normales.
- usa p_expected_version para detectar VERSION_CONFLICT.
- usa idempotencia para evitar duplicados por reintentos.
- valida telefono unico por licencia.
- registra evento customer create/update.
- emite broadcast POS pequeno.

public.pos_delete_customer:
- valida contexto y permiso customers.
- no elimina si debt_cents > 0.
- hace soft delete con deleted_at.
- incrementa server_version.
- registra evento customer delete.
- emite broadcast POS pequeno.

public.pos_pull_customers_snapshot:
- valida contexto.
- devuelve clientes solo de la licencia validada.
- paginacion estable por updated_at asc, id asc.
- soporta include_deleted.

public.pos_pull_customer_changes:
- valida contexto.
- lee pos_sync_events con entity_type customer.
- devuelve events, customers, deleted_ids, latest_change_seq, server_latest_change_seq y has_more.

public.pos_migrate_local_customers:
- migracion batch controlada.
- llama pos_upsert_customer por cada cliente usando llaves idempotentes de migracion.

IDEMPOTENCIA
Se usa public.pos_idempotency_keys con unique (license_id, idempotency_key). Si una operacion completed se reintenta, se devuelve response_payload anterior. Si esta processing, se devuelve IDEMPOTENCY_PROCESSING.

CONFLICTOS
VERSION_CONFLICT se registra en public.pos_sync_conflicts con payload local y payload servidor. DUPLICATE_PHONE devuelve field phone. CUSTOMER_DELETED evita editar clientes eliminados. CUSTOMER_HAS_DEBT evita borrar clientes con deuda.

EVENTOS Y REALTIME
Cada upsert/delete exitoso registra public.pos_sync_events con entity_type customer, entity_id, operation, change_seq, server_version e idempotency_key. El broadcast solo envia entity_type, entity_id, operation, change_seq, server_version, actor ids y created_at. No se manda informacion sensible del cliente. El frontend debe hacer pull oficial.
