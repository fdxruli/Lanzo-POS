INFORME SUPABASE - FASE 6E CANCELACIONES/DEVOLUCIONES CLOUD PRO

Objetivo
Implementar cancelacion cloud segura para ventas PRO confirmadas como source_mode = cloud_committed, revirtiendo efectos reales de caja, inventario/lotes, credito/ledger, eventos sync y reportes sin borrar historial.

Alcance 6E
- Cancelar ventas cloud completas.
- Marcar la venta como cancelled.
- Registrar motivo, actor, staff, dispositivo e idempotency key.
- Revertir caja con movimientos compensatorios.
- Revertir inventario/lotes con movimientos return_in.
- Revertir credito/ledger con movimientos de reversa.
- Recalcular deuda del cliente.
- Emitir eventos sync para venta, cancelacion, caja, inventario, cliente/ledger y reportes.
- Mantener FREE/local intacto.

Fuera de alcance 6E
- No se implementa devolucion parcial.
- No se borra venta ni historial original.
- No se borra ledger original.
- No se borra movimiento de caja original.
- No se borra movimiento de inventario original.
- No se hace migracion/reportes finales de utilidad historica definitiva; queda para Fase 6F.

RPC creada
public.pos_cancel_cloud_sale(...)
- SECURITY DEFINER confirmado.
- Valida licencia, dispositivo y staff.
- Valida feature flag.
- Valida permiso POS y permiso especifico si existe en staff_permissions.
- Bloquea la venta FOR UPDATE.
- Solo permite source_mode = cloud_committed y status = closed.
- Usa idempotencia sales.cloud_cancel:<sale_id>:<device_id>.
- Si la venta ya fue cancelada devuelve SALE_ALREADY_CANCELLED.
- Si la venta no existe devuelve SALE_NOT_FOUND.
- Si la venta no es cloud_committed devuelve SALE_NOT_CLOUD_COMMITTED.

Reversa de caja
- No borra movimientos originales.
- Como pos_cash_movements.amount exige amount > 0, se registran movimientos compensatorios positivos:
  - cancelacion_venta
  - cancelacion_abono_inicial
- Los totales oficiales del turno se decrementan dentro de la misma transaccion.
- Se recalcula expected_cash_total con private.recalculate_pos_cash_session_totals.
- Permite reversa auditada sobre la caja original aunque el turno haya cerrado.

Reversa de inventario/lotes
- Busca movimientos sale_out de la venta.
- Crea movimientos return_in con source = sale_cancellation.
- Para producto sin lote aumenta stock del producto.
- Para producto con lote aumenta stock del lote y recalcula stock del producto desde lotes activos.
- Registra eventos product, product_batch, inventory_movement y sale_item.

Reversa de credito/ledger
- No borra ledger original.
- Si hubo abono inicial, inserta primero CANCEL_PAYMENT positivo para mantener balance_after no negativo.
- Inserta CANCEL_CHARGE negativo para revertir el cargo original.
- Recalcula deuda con private.recalculate_pos_customer_debt.
- Si la deuda quedaria negativa, falla toda la transaccion y no marca venta cancelada.
