1. OBJETIVO
============================================================

Se implemento una fase corta de endurecimiento sobre FASE 6E — Cancelaciones/devoluciones cloud PRO.

Objetivo principal:
- validar preventivamente una cancelacion cloud antes de tocar datos reales;
- mostrar un preview/dry-run de reversas;
- bloquear escenarios riesgosos;
- auditar bloqueos;
- diagnosticar integridad despues de cancelar;
- no romper FREE/local;
- preservar idempotencia y la logica transaccional ya existente de FASE 6E.

2.1 RPC public.pos_preview_cloud_sale_cancellation

Nueva RPC de preview/dry-run.

Firma:
public.pos_preview_cloud_sale_cancellation(
  p_license_key text,
  p_device_fingerprint text,
  p_security_token text default null,
  p_staff_session_token text default null,
  p_sale_id text default null,
  p_reason text default null
) returns jsonb

Comportamiento:
- valida contexto PRO/staff/admin mediante private.validate_pos_sync_context;
- valida feature cloud_sales_cancellations;
- valida permiso POS;
- valida existencia y estado de la venta;
- NO modifica ventas/caja/inventario/credito;
- devuelve can_cancel=true/false;
- devuelve block_reasons con codigos claros;
- devuelve preview de lo que se revertiria en:
  - caja;
  - inventario/lotes;
  - credito/ledger;
- devuelve actor y folio.

Notas:
- La RPC no escribe datos POS de venta, caja, inventario o credito.
- La validacion de contexto puede actualizar metadatos internos de sesion staff, segun la funcion private.validate_pos_sync_context existente, pero no altera la venta ni sus efectos contables.

2.2 Soporte dry-run

Se implemento reutilizando la RPC preview como dry-run seguro.

Ademas, public.pos_cancel_cloud_sale ahora funciona como wrapper preventivo:
1. ejecuta preview;
2. si preview falla o can_cancel=false, NO llama al motor de cancelacion;
3. registra auditoria sale.cancel_blocked cuando aplica;
4. devuelve success=false con block_reasons;
5. solo si preview permite cancelar, llama al motor transaccional 6E.

2.3 Wrapper public.pos_cancel_cloud_sale

La funcion original de FASE 6E fue renombrada a:
- public.pos_cancel_cloud_sale_apply_fase6e(...)

La nueva public.pos_cancel_cloud_sale(...) conserva la misma firma que consumia el frontend, pero ahora agrega la capa preventiva.

Esto evita romper el frontend existente y mantiene la idempotencia del motor original.

2.4 RPC public.pos_validate_cloud_sale_integrity

Nueva RPC de diagnostico post-cancelacion.

Firma:
public.pos_validate_cloud_sale_integrity(
  p_license_key text,
  p_device_fingerprint text,
  p_security_token text default null,
  p_staff_session_token text default null,
  p_sale_id text default null
) returns jsonb

Valida:
- venta cloud_committed;
- estado cancelled/cancelled_at;
- registro unico en pos_sale_cancellations;
- reversas de caja contra movimientos originales;
- return_in de inventario contra sale_out original;
- reversas de ledger CANCEL_CHARGE/CANCEL_PAYMENT;
- deuda del cliente contra suma del ledger;
- abonos posteriores independientes;
- eventos sync de sale, sale_cancellation y report.

Devuelve:
- success;
- is_valid;
- code;
- issues[];
- summary.cash;
- summary.inventory;
- summary.credit;
- summary.sync.

2.5 Bloqueo de ventas fiadas con abonos posteriores independientes

Se agrego bloqueo preventivo:
- SALE_HAS_SUBSEQUENT_CUSTOMER_PAYMENTS

Criterio:
- venta con credit_effect_status aplicado;
- existen movimientos pos_customer_ledger del mismo cliente:
  - type = PAYMENT;
  - amount < 0;
  - sale_id distinto a la venta a cancelar;
  - no son reversas;
  - creados despues del cargo de la venta.

Resultado:
- no se modifica caja;
- no se modifica inventario;
- no se modifica ledger/deuda;
- no se marca la venta como cancelada;
- se devuelve bloqueo claro.

2.6 Auditoria sale.cancel_blocked

Cuando public.pos_cancel_cloud_sale bloquea una cancelacion por preview, registra:
- event_type = sale.cancel_blocked

Payload incluye:
- sale_id;
- folio;
- reason;
- block_code;
- block_reasons;
- preview;
- idempotency_key;
- phase = fase6e1_hardening_cancelaciones_cloud.

============================================================
5. IDEMPOTENCIA
============================================================

Se mantiene idempotencia del motor original de FASE 6E.

El wrapper preventivo no inserta idempotency processing antes de preview.
Solo si preview permite la cancelacion, delega a public.pos_cancel_cloud_sale_apply_fase6e(...), que conserva la logica existente de pos_idempotency_keys.

En bloqueos:
- no se crea reversa;
- no se marca venta como cancelada;
- no se modifica caja/inventario/credito;
- solo se registra auditoria sale.cancel_blocked cuando existe venta y contexto valido.
