1. OBJETIVO

Implementar la FASE 6C para que las ventas PRO confirmadas en cloud puedan descontar inventario real en Supabase de forma transaccional, idempotente y auditable.

La prioridad aplicada fue:

1) No vender sin stock cloud suficiente.
2) No descontar inventario dos veces.
3) No mover caja si inventario falla.
4) No tocar credito/ledger en esta fase.
5) Mantener FREE/local estable.

2. ALCANCE

Esta fase agrega inventario cloud a la venta efectiva cloud de Fase 6B.

La ruta 6B existente se conserva mediante public.pos_create_cloud_sale_cashier.
La ruta 6C usa una RPC nueva: public.pos_create_cloud_sale_cashier_inventory.

Esto reduce riesgo porque permite mantener estable la venta cloud con caja/folio sin inventario, y activar inventario solo cuando el feature flag cloud_sales_inventory este encendido.

3. QUE SI HACE 6C

- Valida stock cloud antes de confirmar una venta cloud con inventario.
- Descuenta stock en public.pos_products para productos sin lotes.
- Descuenta stock en public.pos_product_batches para productos con lotes.
- Resuelve lotes en este orden:
  1. batch_id explicito.
  2. batches_used / batchesUsed.
  3. FEFO por expiry_date mas proxima y luego created_at.
- Bloquea filas con FOR UPDATE durante validacion/aplicacion.
- Registra movimientos en public.pos_inventory_movements.
- Marca pos_sales.inventory_effect_status como applied o not_required.
- Marca pos_sale_items.inventory_effect_status y datos before/after.
- Emite eventos sync para sale, sale_item, product, product_batch, inventory_movement y report.
- Registra auditoria en pos_sale_audit_events.
- Mantiene idempotencia por venta y por movimiento de inventario.
- Refresca catalogo local por pull incremental despues de una venta 6C.
- Mantiene Dexie como cache, no como fuente oficial.

4. QUE NO HACE 6C

- No crea pos_customer_ledger.
- No modifica deuda de cliente.
- No permite venta fiada cloud.
- No implementa cancelaciones cloud.
- No implementa devoluciones cloud.
- No devuelve stock por cancelacion.
- No toca FREE/local.
- No usa stock local como fuente oficial para confirmar venta cloud inventory.

7. RPCS / HELPERS

Nueva RPC publica:

public.pos_create_cloud_sale_cashier_inventory(...)

Esta RPC:

1. Valida licencia/dispositivo/staff.
2. Valida feature flags 6C.
3. Valida permiso POS.
4. Inserta idempotencia principal.
5. Prevalida inventario con bloqueos FOR UPDATE.
6. Si inventario falla, responde success=false y no crea venta/caja/folio.
7. Si inventario pasa, llama public.pos_create_cloud_sale_cashier con idempotency key interna.
8. Aplica inventario en la misma transaccion.
9. Registra movimientos/auditoria/eventos.
10. Devuelve response final con venta, items, pagos, caja e inventario.

Helpers privados creados:

- private.assert_cloud_sales_inventory_enabled(context jsonb)
- private.product_uses_batches(product)
- private.normalize_sale_inventory_item(...)
- private.resolve_sale_inventory_allocations(...)
- private.apply_sale_inventory_effects(...)
- private.record_pos_inventory_movement(...)
- private.pos_inventory_movement_to_jsonb(...)

8. LOGICA PRODUCTO SIN LOTE

Para productos con track_stock=true y sin manejo de lotes:

1. Bloquea public.pos_products con FOR UPDATE.
2. Valida licencia, existencia, activo y no eliminado.
3. Calcula disponible = stock - committed_stock.
4. Si disponible < quantity, devuelve INSUFFICIENT_CLOUD_STOCK.
5. Si hay stock, descuenta stock.
6. Actualiza updated_at/server_version.
7. Registra pos_inventory_movements con movement_type=sale_out/source=sale.
8. Emite evento sync product.
9. Actualiza pos_sale_items con stock_before/stock_after.

9. LOGICA PRODUCTOS CON LOTES

Para productos con batch_management o item con batch_id/batches_used:

1. Bloquea producto con FOR UPDATE.
2. Resuelve lote explicito, asignaciones o FEFO.
3. Bloquea lotes con FOR UPDATE.
4. Valida licencia/producto/lote activo/no eliminado.
5. Valida stock suficiente en lote(s).
6. Descuenta public.pos_product_batches.stock.
7. Recalcula stock agregado de public.pos_products desde lotes activos.
8. Actualiza server_version de lote y producto.
9. Registra un movimiento de inventario por lote usado.
10. Emite eventos sync product_batch/product/inventory_movement.

10. FEFO

Si el item no trae batch_id ni batches_used, y el producto usa lotes, el backend aplica FEFO:

1. Lotes activos.
2. Lotes no eliminados.
3. Lotes con stock disponible.
4. Orden por expiry_date asc nulls last.
5. Luego created_at asc.
6. Luego id asc.

11. IDEMPOTENCIA

La RPC 6C usa idempotencia de dos niveles:

- Key externa: sales.cloud_commit.inventory:<saleId>:<deviceId>
- Key interna 6B core: <key externa>:cashier_core

Los movimientos de inventario usan keys derivadas:

<idempotency_key>:inventory:<sale_item_id>:<product_id>:<batch_id>:<ordinal>

Esto protege contra:

- doble click;
- retry de red;
- reenvio de la misma venta;
- doble descuento de inventario;
- doble movimiento de caja.

12. EVENTOS SYNC

Se registran eventos para:

- sale
- sale_item
- sale_payment
- cash_session
- cash_movement
- product
- product_batch
- inventory_movement
- report

Frontend agrega INVENTORY_MOVEMENT a SYNC_ENTITY_TYPES.
Producto/catalogo escucha inventory_movement para disparar pull de catalogo.
Ventas cloud tambien escucha inventory_movement como señal complementaria.

Realtime sigue siendo aviso; pull incremental sigue siendo fuente oficial.

18. ESTADO FINAL ESPERADO

Con la fase 6C activa en PRO:

- Venta cloud confirma caja/folio/inventario.
- Venta en efectivo mueve caja e inventario.
- Venta tarjeta/transferencia descuenta inventario sin mover efectivo.
- Producto sin lote descuenta pos_products.stock.
- Producto con lote descuenta pos_product_batches.stock.
- Stock insuficiente rechaza la venta completa.
- No se toca credito/ledger.
- No se implementan cancelaciones/devoluciones.
- FREE/local sigue intacto.
