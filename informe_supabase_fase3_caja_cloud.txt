1. OBJETIVO
Se implementó la base cloud de caja PRO con modelo online-first, fuente de verdad en Supabase, Dexie como caché local, realtime como aviso y pull incremental como mecanismo oficial de actualización.

2. FEATURE FLAGS
Se agregó cloud_cash_sync al JSON features de public.plans:
- pro_monthly: cloud_cash_sync = true
- free_trial: cloud_cash_sync = false
- basic_monthly: cloud_cash_sync = false

La caja cloud exige dos flags activos:
- cloud_pos_sync = true
- cloud_cash_sync = true

6. MODELO DE ACTOR
actor_key:
- staff: staff:<staff_user_id>
- admin: admin_device:<device_id>

Esto evita dos cajas abiertas para el mismo actor y permite cajas simultáneas separadas:
- admin en su dispositivo
- staff A
- staff B

7. RPCs PÚBLICAS
Todas las RPCs son security definer y validan licencia/dispositivo/staff.

7.1 public.pos_get_current_cash_session
Devuelve la caja abierta del actor actual, movimientos y resumen admin si aplica.

7.2 public.pos_open_cash_session
Abre caja cloud para el actor actual.
Reglas:
- staff requiere permiso cash_register/caja
- staff ignora responsible_name enviado y usa display_name/username
- admin default Administrador, con override permitido
- idempotente
- evita doble caja abierta por actor

7.3 public.pos_register_cash_movement
Registra entrada/salida/ajuste.
Reglas:
- caja debe existir y estar abierta
- staff solo opera su propia caja
- salida no puede dejar caja negativa
- recalcula totales en servidor
- registra auditoría y sync event

7.4 public.pos_adjust_initial_cash_fund
Ajusta fondo inicial con expected_version.
Reglas:
- caja abierta
- staff solo su caja
- version conflict devuelve VERSION_CONFLICT
- no-op seguro si el monto es igual

7.5 public.pos_close_cash_session
Cierra caja con conteo físico.
Reglas:
- conteo físico >= 0
- fondo siguiente turno >= 0
- fondo siguiente turno <= conteo físico
- recalcula totales oficiales dentro de transacción
- calcula diferencia en servidor
- marca caja cerrada

7.6 public.pos_pull_cash_snapshot
Devuelve snapshot de sesiones y movimientos.
Scopes:
- mine
- open
- all
- staff
Staff normal queda forzado a mine.

7.7 public.pos_pull_cash_changes
Devuelve cambios incrementales para cash_session y cash_movement.
Staff solo recibe cambios propios.
Admin recibe cambios de la licencia.

7.8 public.pos_admin_list_cash_sessions
Lista cajas para auditoría admin.
Filtros:
- status
- staff_user_id
- date_from
- date_to
- limit
- offset

7.9 public.pos_admin_get_cash_session_detail
Devuelve detalle de una caja, movimientos y eventos de auditoría.

8. SYNC EVENTS Y REALTIME
Se usan public.pos_sync_events.
Entity types:
- cash_session
- cash_movement

Operations:
- open
- close
- update
- movement
- adjust

Broadcast usa private.broadcast_pos_event y solo envía payload seguro:
- entity_type
- entity_id
- operation
- change_seq
- server_version
- actor_device_id
- actor_staff_user_id
- created_at

No se envían montos completos por broadcast.

9. IDEMPOTENCIA
Se usa public.pos_idempotency_keys mediante helpers existentes.
Operation types:
- cash.open
- cash.close
- cash.movement
- cash.adjust_initial_fund

Reglas:
- misma key completada devuelve la misma respuesta
- misma key en proceso devuelve respuesta segura
- no duplica movimientos

10. CONCURRENCIA
- Dos aperturas simultáneas se resuelven con unique parcial por actor abierto.
- Salidas se validan dentro del RPC con la caja bloqueada y totales recalculados.
- Cierre y ajuste usan expected_version.
- VERSION_CONFLICT obliga a refrescar.

11. PERMISOS
- Admin puede operar y auditar.
- Staff requiere cash_register o legacy caja para operar caja.
- Auditoría para staff solo si tiene reports, cash_audit o caja_auditoria.
- Staff sin permiso no puede abrir ni mover caja por RPC.

13. ONLINE-FIRST
En PRO cloud no se encolan movimientos de dinero.
Si no hay internet:
- no abrir caja
- no cerrar caja
- no entrada
- no salida
- no ajuste
- se muestra último estado cacheado cuando existe
- se muestra mensaje claro de conexión

14. QUÉ QUEDA PARA VENTAS CLOUD
- Registrar venta_efectivo en pos_cash_movements dentro del mismo RPC transaccional de venta cloud.
- Ligar sale_id/reference_id a cash_session_id.
- Evitar ventas cash si caja cloud no está abierta.

15. QUÉ QUEDA PARA ABONOS CLOUD
- Crear RPC transaccional para abono_cliente que actualice deuda y caja en una misma operación.
- No mezclar abonos locales con caja cloud.

16. QUÉ QUEDA PARA REPORTES CLOUD
- Reportes consolidados por licencia/staff/dispositivo/turno.
- Dashboard admin cloud con filtros avanzados.
- Exportación server-side opcional.
